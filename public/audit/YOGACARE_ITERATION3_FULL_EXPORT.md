# SECTION E — DIFFERENTIATION SELF-AUDIT

### 1. Does the homepage visually show adaptation, or just state it?
**Mostly states it.** The TailoringSnapshot component on the Plan page visually shows adaptation (chips for duration, energy, time, flare, condition). But the homepage itself only describes adaptation through copy — there is no dynamic visual proof on the landing page. The signal bar is static.

### 2. Does it feel distinct from Down Dog?
**Partially.** The dark ink hero with gold accents is distinct. The medical safety-first positioning (disclaimer, red flags, STOP page) is unique. The condition-specific questionnaire is more clinical than Down Dog. However, the exercise card layout and session flow are structurally similar to most yoga apps.

### 3. Does the hero feel empty?
**No.** The AdaptiveWaveHero SVG provides subtle visual depth with animated gold/sage curves and a spine motif. The dark-on-ink contrast is premium. It does not feel empty but could benefit from more visual complexity or motion.

### 4. Does it feel premium or sparse?
**Premium-leaning.** The Plus Jakarta Sans typography, generous whitespace, card-premium styling, and consistent gold/sage/ink palette create a premium feel. Some pages (Conditions, Setup) feel slightly form-heavy but not sparse.

### 5. Are there inconsistencies?
**Yes:**
- ExerciseVideoAnimation and ExerciseAnimationV7 use hardcoded Tailwind color classes (blue-50, emerald-50, amber-50, purple-50) instead of design tokens
- HeroSection uses inline HSL values `bg-[hsl(var(--color-ink))]` and `from-[hsl(215,50%,11%)]` instead of semantic tokens
- The ExpertReview page is entirely in Hebrew with RTL layout (`dir="rtl"`)
- MasterExercises data (titles, instructions, breathing cues, safety, why) remains in Hebrew — only the UI chrome is English
- Settings page uses `yaelYogaDisableAnimations` localStorage key (legacy naming)
- Animation V7 uses hardcoded colors for SKIN, HAIR, TOP, BOTTOM instead of CSS variables (V8's HumanFigure does use CSS vars)

### 6. Are there placeholder visuals?
**Yes:**
- PageIllustration components are minimal SVG ellipses with Lucide icons — functional but placeholder-quality
- No real exercise demonstration videos exist (breath-demo.mp4 and mobility-demo.mp4 are category-level demos, not per-exercise)
- The YC monogram in "Guided by Experience" is a text placeholder, not a designed mark

## F.6 Plan Generation Algorithm

### Mode Determination (determineModeV2)
```
if profile.flareToday → "flare"
if flareNow === "yes" → "flare"
if pain >= 7 OR fatigue >= 7 OR sleep <= 3 → "flare"
if pain >= 5 OR fatigue >= 5 → "easier"
if energyLevel === "low" → "easier"
else → "normal"
```

### Exercise Count by Duration
| Minutes | Count |
|---------|-------|
| ≤10 | 3 |
| ≤15 | 5 |
| ≤20 | 6 |
| ≤30 | 8 |
| ≤45 | 10 |
| >45 | 12 |

### Category Balance (Normal Mode)
- 1 breath
- 25% stability (min 2)
- 20% release (min 1, or 30% min 2 if sleep_issues)
- Remainder → mobility (min 2)

### Scoring Pipeline
1. For each exercise: calculate relevance score per condition (weighted by CONDITION_WEIGHT)
2. Primary condition score × 2, secondary conditions averaged
3. Mode modifiers: flare → +2 breath/release, -2 stability; easier → +1 breath, +0.5 release
4. Stress/fibro/sleep bonus: +1 for breath and release
5. Recent usage penalty: -2
6. No-equipment bonus: +0.3
7. Seeded shuffle of top half for deterministic variety

### Session Distribution
| Sessions/Week | Day Indices |
|---------------|-------------|
| 2 | [2, 5] |
| 3 | [2, 4, 6] |
| 4 | [1, 3, 5, 7] |
| 5 | [1, 2, 4, 6, 7] |

## F.7 Safety Stop Logic
If ANY red flag is checked in questionnaire step 3 → `navigate("/stop")`

Red flags:
1. New sharp pain
2. Increasing numbness or weakness
3. Unusual dizziness or fainting
4. Shortness of breath
5. Fever or acute illness

## F.8 Check-In Adaptive Logic (adaptNextSession)
Triggers adaptation if: `tooMuch === true` OR `painAfter - painBefore >= 2`
- Finds next planned session
- Sets mode to: flare (if flareToday) or easier
- Re-generates exercise IDs using selectExercisesForConditions with new mode

## F.9 Anti-Repetition System
- Tracks last 20 used exercise IDs in localStorage (`yogacareRecentExercises`)
- Recent exercises get -2 score penalty
- Seeded shuffle uses week number + condition string hash for deterministic variety per week

# SECTION I — DIFFERENTIATION STATEMENT (CURRENT)

## How YogaCare CURRENTLY differentiates from competitors:

1. **Condition-specific safety filtering (vs Generic Yoga Apps / Down Dog):** Every exercise has structured safety tags (`pregnancySafe`, `discSafe`, `kneeSafe`, `oaSafe`, `shoulderSafe`, `flareSafe`) and contraindication flags. The scoring engine filters and ranks exercises per-condition with weighted relevance scores. Down Dog uses style/difficulty preferences but not medical condition filtering.

2. **Adaptive check-in loop (vs Peloton Yoga):** The pain/fatigue/sleep check-in triggers real-time mode adaptation (normal → easier → flare). The `adaptNextSession` function adjusts the next session's mode and exercise selection based on post-session feedback. Peloton is instructor-led with no adaptive feedback loop.

3. **Red flag safety system (vs Meditation Apps):** Clinical-grade red flag detection routes users to a STOP page, preventing exercise during acute conditions. No meditation or generic yoga app implements this level of medical safety gating.

4. **Multi-condition scoring with primary weighting (vs Physical Rehab Apps):** Users can select multiple conditions simultaneously. The scoring engine weights the primary condition 2x and averages secondary conditions, producing nuanced exercise selection. Traditional PT apps typically address single conditions.

5. **Deterministic variety with anti-repetition (vs All):** The seeded shuffle algorithm produces different but deterministic exercise selections per week, while the recent-usage tracking prevents the same exercises from appearing in consecutive sessions. This is a structural differentiation not found in other adaptive systems.
